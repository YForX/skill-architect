# 三路方案对比模板

以下模板用于阶段 ③，确保三种方案的输出格式一致、可比。

---

## 方案摘要模板

对方案 A（拿来）、B（融合）、C（原生）各输出以下内容：

```markdown
### 方案 [A/B/C]：[方案名称]

**核心思路**：[一句话概括这个方案做什么、怎么做]

**关键设计决策**：
- [决策 1]：[为什么这样选]
- [决策 2]：[为什么这样选]
- [决策 3]：[为什么这样选]（可选，2-3 条即可）

**预计结构**：
​```
skill-name/
├── SKILL.md          ← [简述里面放什么]
├── scripts/          ← [简述需要什么脚本]（如不需要则标「不需要」）
├── references/       ← [简述需要什么参考文件]（如不需要则标「不需要」）
└── assets/           ← [简述需要什么资源]（如不需要则标「不需要」）
​```

**优势**：
- [优势 1]
- [优势 2]

**劣势**：
- [劣势 1]
- [劣势 2]

**预计工作量**：[小 / 中 / 大] — [简要说明]
​```
```

## 方案对比矩阵模板

```markdown
### 方案对比矩阵

| 维度 | 方案 A: [名称] | 方案 B: [名称] | 方案 C: [名称] |
|------|:-----------:|:-----------:|:-----------:|
| 开发成本 | [低/中/高] | [低/中/高] | [低/中/高] |
| 基础来源 | 基于 [已有Skill名] | 融合 [N] 个 Skill | 从零设计 |
| 可维护性 | [依赖上游更新] 或 [自主可控] | [中] | [自主可控] |
| 创新程度 | [低：沿用现有思路] | [中：重组现有思路] | [高：全新架构] |
| 需求适配度 | [中：需要妥协部分需求] | [高：取各家之长] | [最高：完全定制] |
| 主要风险 | [上游停更/不兼容] | [整合复杂度] | [从头造轮子] |
| 核心收益 | [描述] | [描述] | [描述] |

**推荐**：[方案 X]

**推荐理由**：[为什么在当下场景中这个方案最合适——考虑需求的特殊性、时间约束、可维护性需求等]
```

---

## 三种方案的区分指南

### 方案 A：拿来主义

**不是**「下载一个 Skill 直接用」——而是「基于最好的已有 Skill，明确改什么」。

必须说明：
- **基础选择**：基于哪个 Skill？为什么选它？
- **保留项**：这个 Skill 的哪些部分直接用，不需要改
- **修改项**：哪些部分需要改？怎么改？
- **删除项**：哪些部分需要去掉？
- **新增项**：哪些功能原 Skill 没有，需要新增？

### 方案 B：融合创新

**不是**「把两个 Skill 拼在一起」——而是「吸收多个来源的优点，设计一个新架构」。

必须说明：
- **融合来源**：吸收了哪几个 Skill（或跨领域模式）的哪些优点？
- **取舍逻辑**：为什么选这些而不选那些？
- **冲突解决**：不同来源之间有矛盾吗？怎么取舍？
- **协同效应**：融合后产生了什么 1+1>2 的效果？

### 方案 C：原生设计

**不是**「忽略调研结果自己瞎写」——而是「调研后发现现有方案都有根本性缺陷，需要新思路」。

必须说明：
- **为什么现有方案不够**：A 和 B 为什么不行？根本性缺陷是什么？
- **新思路的来源**：虽然不是直接基于某个 Skill，但思路从哪里来？（跨领域启发、第一性原理、用户独特需求）
- **与调研的关联**：即使从零设计，调研中的哪些发现影响了设计？

---

## 输出注意事项

1. **保持摘要级**：每个方案 5-10 行正文，不要写成完整的 SKILL.md
2. **可比较性**：三个方案应该在同一抽象层级，方便用户横向对比
3. **不隐藏偏好**：给出明确推荐，但标注哪个是推荐、哪些是备选
4. **引用调研**：方案中的设计决策应该能回溯到阶段 ② 的具体发现
5. **诚实说明不确定之处**：如果有不确定的设计选择，标注出来让用户判断
