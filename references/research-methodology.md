# 竞品调研方法论

## 搜索策略

### 第一轮：同领域搜索

**目标**：找到直接竞品 Skill，了解同行的做法。

**步骤**：

1. **提取核心关键词**
   - 从用户需求中提取 2-3 个核心概念
   - 例如「帮我做一个生成 PPT 的 Skill」→ 关键词：`ppt`、`presentation`、`slide`

2. **多工具并行搜索**
   ```bash
   # 工具 1：npx skills（按安装量排序）
   npx skills find "ppt presentation"

   # 工具 2：findskill（按 GitHub star 排序）
   findskill search "presentation slide generation"

   # 工具 3：WebSearch（兜底）
   ```

   搜索词变化策略：
   - 英文词 + 中文词各搜一次
   - 用同义词替换（如 `pdf` ↔ `document` ↔ `file processing`）
   - 用「用途」而非「技术」搜（如搜 `generate slides` 而不是 `python-pptx skill`）

3. **从搜索结果中筛选**
   - 排除安装量 < 100 或 star < 50 的（质量太差参考价值低）
   - 排除最后更新超过 1 年的（可能过时）
   - 优先保留名字和描述最匹配的 2-5 个
   - 如果匹配的不足 2 个，扩大搜索词范围；超过 5 个，取安装量/star 最高的 5 个

### 第二轮：跨领域搜索

**目标**：找到不同领域但有可迁移设计模式的 Skill。

**方法**：不是随机搜，而是**先确定要迁移什么模式，再搜对应的领域**。

**模式识别指南**：

| 如果你的 Skill 涉及 | 可跨领域搜索的模式 | 示例目标领域 |
|---------------------|-------------------|-------------|
| 多步骤处理管线 | 管线编排、降级策略、错误恢复 | 图像处理、ETL 数据处理 |
| 表单/文档填写 | 模板填充、字段映射、验证 | PDF 表单、HTML 模板引擎 |
| 数据分析/报告 | 数据提取 → 分析 → 可视化 → 叙述 | 财务分析、BI 报表 |
| 代码生成/审查 | 检查清单、严重等级、修复建议 | 安全审计、医疗诊断 |
| 搜索/检索 | 多源聚合、排序、去重 | 研究助理、新闻聚合 |
| 格式转换 | 解析 → 中间表示 → 输出 | 文档转换、数据迁移 |
| 交互式引导 | 状态机、分步确认、回退 | 向导系统、教学 Skill |

**搜索方法**：
- 确定 1-2 个要迁移的设计模式
- 搜索该模式在**其他领域的典型应用**
- 不限于 Skill——也可以搜开源工具、SaaS 产品的设计

## 评估维度

对每个调研到的 Skill，从以下维度打分（1-5）：

### 功能维度
- **覆盖度**：它能覆盖我们需求的百分之多少？
- **正确性**：核心功能做得对不对？（看源码/文档，不只看描述）
- **边界处理**：edge case 处理得好吗？

### 设计维度
- **架构清晰度**：文件组织、指令分层是否合理？
- **可扩展性**：容易在基础上修改/扩展吗？
- **可维护性**：代码质量、文档质量如何？

### 生态维度
- **活跃度**：最近有更新吗？issue/PR 活跃吗？
- **社区接受度**：安装量/star 能说明多少人真的在用
- **依赖健康度**：依赖是否合理？有没有过度依赖？

## 质量标准

### 值得纳入调研的 Skill

- 安装量 100+ 或 GitHub star 50+
- 有实质性的 SKILL.md（不是只有 5 行）
- 解决的是真实问题（不是 demo/toy）

### 不值得纳入的

- 安装量 < 50 且无 star
- 最后更新超过 18 个月且功能明显过时
- 描述和实际内容不符（标题党）
- 有明显安全风险或恶意代码嫌疑

### 调研深度判断

根据需求复杂度调整：

| 需求复杂度 | 调研深度 | 每个 Skill 分析时间 |
|-----------|---------|-------------------|
| 简单 | 读描述 + SKILL.md 概览 | 2-3 分钟 |
| 中等 | 读完整 SKILL.md + 目录结构 | 5-8 分钟 |
| 复杂 | 读 SKILL.md + 参考文件 + 脚本 | 10-15 分钟 |

## 可迁移模式的识别

这是调研最重要的产出。不要只停留在「这个 Skill 做了什么」，要追问：

1. **它怎么组织信息的？** — 指令放在 SKILL.md 还是 references/？脚本粒度如何？
2. **它怎么处理错误？** — 有降级策略吗？有验证脚本吗？
3. **它怎么引导 Claude 的行为？** — 用强制指令还是解释推理？用示例还是规则？
4. **它怎么管理复杂度？** — 多领域如何分层？长文件如何组织？
5. **它的触发描述怎么写？** — 为什么它能被正确触发？

**好的可迁移发现示例**：
> 「财务分析 Skill 用了一个巧妙的分层：SKILL.md 只定义分析框架（收入/支出/趋势），具体的会计准则放在 references/ 里按需加载。我们的 PPT Skill 也可以这样做——SKILL.md 定义幻灯片结构框架，具体的配色/字体/版式放在 references/ 里。」
